## Staleness Flags

### `tests/agent-harness/e2e-tests.sh`
- **Section 11** (`Message send/receive`): Uses `$HEX_AGENT message` (old subcommand path) then verifies via `$HEX_BIN message list`. The agent subcommand wraps the binary but the test is inconsistent — it should use the unified binary directly.
- **Section 12** (`Messaging primitive`): Uses `--msg-type` flag. The current binary CLI uses `--type` (verify by checking `main.rs` MessageCommands::Send). **Potential staleness.**
- References `$HEX_DIR/.hex/messages` in directory setup (Dockerfile line: `$HOME/hex/.hex/messages`). This is the legacy JSONL storage path; the system now uses SQLite. The directory setup may be cargo-culted from before the migration.
- Dockerfile only installs `bash` — no SQLite CLI, no Python, cannot test database contents directly.

### `tests/test_e2e.sh`
- Tests `bash .hex/scripts/doctor.sh` directly rather than `hex doctor` (Rust binary). If the shell script is deprecated in favor of the unified binary, this test tests stale behavior.
- Tests `bash .hex/scripts/startup.sh` — similarly, may be replaced by `hex server start`.
- Memory tests call Python scripts directly (`memory_save.py`, `memory_search.py`), not `hex memory` CLI commands.
- Upgrade test creates a local git repo — fragile, depends on git being available in CI.
- Uses `/tmp/test-hex` as install path — not isolated in Docker.

### `tests/test_fullstack.sh`
- Describes testing `hex-events` Python daemon integration — but the spec says hex-events is now a Rust binary subcommand (`hex events`). Tests may be testing the archived Python implementation.
- References `hex-events` as a separate process, which contradicts the unified binary model in v0.8.0.

### `tests/events/*.py`
- Unit tests for the Python event engine. If the event engine is now compiled into the Rust binary, these tests cover a potentially archived implementation. Verify against `system/harness/src/events.rs`.
- These are not containerized E2E tests — they run in-process with mocked state.

### `tests/migrate/test-migrate.sh`
- Tests v1→v2 directory layout migration. Does not test comments.json→SQLite or messages/*.jsonl→SQLite data migration (the new SQLite unification in v0.8.0).

## Recommended Suite Structure for `tests/core-e2e/`

Based on this audit, the new `core-e2e` suite should:

1. **Use the compiled Rust binary exclusively** — no shell scripts, no Python memory scripts
2. **Verify SQLite state directly** — use `sqlite3` CLI to query `.hex/data/hex.db` after operations
3. **Replace OR-pattern assertions** with specific column/field checks
4. **Add version consistency test** — `hex version` output must match `version.txt`
5. **Test all 12 subcommand groups** end-to-end via the unified `hex` binary
6. **Include server lifecycle** — start server, test SSE, stop cleanly

### Priority order (highest coverage gap first):
1. CLI subcommand accessibility (cli) — verifies the binary is wired correctly
2. Messaging (messaging) — partial coverage exists, needs strengthening
3. Events CLI (events) — zero coverage via CLI
4. Asset registry (assets) — zero coverage
5. SSE bus (sse) — zero coverage
6. Telemetry verification (telemetry) — zero CLI coverage

### What the suites verify

**`test-boi-install.sh`** — Fresh-container install:
- `~/.boi/bin/boi` is a real executable (symlink target also verified)
- `boi --help` exits 0 and lists `dispatch`, `status`, `bench`, `cancel`
- `boi version` output matches VERSIONS `BOI_VERSION`
- Wrapper chain `boi.sh --help` exits 0
- (With `ANTHROPIC_API_KEY`) full smoke dispatch completes and marker file exists

**`test-boi-upgrade.sh`** — Upgrade from prior release:
- Baseline install captures version + binary mtime
- After checkout-HEAD + `install.sh` re-run: version bumped, binary mtime newer
- `boi --help` lists all subcommands from new version (dynamically checked)
- Smoke dispatch works after upgrade
- BAD case: corrupted symlink is caught by `doctor.sh` runtime check
